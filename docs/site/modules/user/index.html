👤 USER Module

User Story Development Guide for admin-level user management flows

📋 Overview

Purpose: Provide admins with full control over system users (list, add, update, delete) while preserving the original COBOL-inspired workflow, keyboard shortcuts and audit messaging.

Business Context: Exposed only via the admin menu (`getAdminMenuData`) and guarded by `ProtectedRoute` (role = admin). Critical for compliance workflows that need quick role adjustments, secure creations, and intentional deletions.

Screening Notes: Each page reuses `SystemHeader`, `LoadingSpinner`, `Alert`, and Material-UI controls along with MSW mocks (`/api/users/*`) before switching to the backend real contracts (`/users/*`).

Last update: March 6, 2026

🎯 User Story Development Guide

📝 Specific US Templates

  • User audits: As an administrator, I want to list users filtered by ID and quickly select Update/Delete so I can validate privileges before taking action.
  • User provisioning: As an admin, I want to create a user with an 8-character ID/password and explicit type (A/U) so that access aligns with policy.
  • User lifecycle: As a compliance officer, I want to fetch a user, edit their role, confirm via F5, and optionally delete them with confirmation so I can keep audit trails clean.

⚡ Development Acceleration Factors

  • `useUserList` / `useUserAdd` / `useUserUpdate` / `useUserDelete`: Encapsulate API calls (`apiClient`, `UserApiAdapter`), loading/error states and keyboard shortcuts, enabling new screens to reuse the same flows.
  • SystemHeader + Alerts: Consistent UI shell with transaction IDs (CU00—CU03) keeps operations traceable and reduces copy/paste.
  • Material-UI Table + Select: Selection-based actions reduce custom input logic, and 8-char limits/new validations are enforced inside `handleFieldChange` helpers.
  • MSW Handlers: `userListHandlers`, `userAddHandlers`, etc., mirror backend responses—ideal for story-driven regression tests.

📊 Complexity Guidelines

  • Simple (1-2 pts): List users with search, pagination (F7/F8) and ENTER=continue selection.
  • Medium (3-5 pts): Implement Add/Update forms with synchronous validations, inline feedback, and F-key shortcuts (F3 back, F4 clear, F5 save).
  • Complex (5-8 pts): Coordinate backend adaptors + MSW responses when switching from mocks to production, ensuring flows honor exact-length password rules and deletion confirmations.

🔧 Technical Foundation

🏗️ Key Components

  • `UserListScreen.tsx`: Material-UI table with selection dropdowns, helper alert area, keyboard listener handling ENTER/F3/F7/F8, and pagination chips.
  • `UserAdd/Update/DeleteScreen.tsx`: Responsive grids, validation helpers, confirmation dialogs, and shared actions hooking into `useUserAdd`, `useUserUpdate`, `useUserDelete`.
  • `SystemHeader` + `LoadingSpinner` + `Alert` components: Provide consistent UX and state messaging across the module.
  • `UserApiAdapter` + `apiClient`: Adapt between `/api/users/` mocks and `/users` real endpoints, handle HTTP errors, and keep auth header management centralized.

🔗 Dependencies

  • Authentication (Auth module): Provides role info + `auth-token` for headers.
  • Redux + ProtectedRoute: Ensures `/admin/users/*` is only accessible to admins.
  • MSW mock servers: `userListHandlers`, `userAddHandlers`, `userUpdateHandlers`, `userDeleteHandlers` used when `VITE_USE_MOCKS=true`.
  • Material UI 5: Layout, tables, forms, dialogs, chips, select inputs.
