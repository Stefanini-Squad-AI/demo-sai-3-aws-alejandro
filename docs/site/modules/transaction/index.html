💸 TRANSACTION - Guía Operacional

Registro, consulta y reportes de transacciones manuales con validación legacy, atajos de teclado y reportes confirmados.

Documentación extendida: docs/modules/transaction/transaction-overview.md

📋 Visión general del módulo

TRANSACTION agrupa cuatro pantallas protegidas (Alta, Lista, Consulta y Reportes) que comparten SystemHeader, hooks cruzados y la capa HTTP via useMutation + apiClient. Las interfaces replican atajos del mainframe (F3=salir, F5/F7/F8, ENTER) y validaciones del backend heredado como confirmación explícita (Y/N) en cada acción crítica.

  • Alta: formulario en pasos (datos + confirmación) que valida account/card, códigos de tipo/categoría y límites de ±99,999,999.99 antes de llamar a POST /api/transactions.
  • Lista: tabla paginada con navegación F7/F8, selección S para ir al detalle y búsqueda rápida por Transaction ID.
  • Consulta: carga la información header (fecha, programa, transactionName) y detalles del merchant tras limpiar el estado con /api/transaction/clear.
  • Reportes: mensual/anual/personalizados con confirmación Y/N, descarga simulada y totales por accountGroups.

🎯 Guía de User Stories

Simple (1-2 pts)

Como operador back-office, quiero ingresar un Transaction ID numérico y presionar ENTER para que la lista muestre los cinco movimientos más recientes y seleccione uno con S.

Medio (3-5 pts)

Como auditor, quiero registrar una transacción manual con merchant completo y confirmación Y para corregir ajustes del mainframe asegurando validaciones de monto y códigos.

Complejo (5-8 pts)

Como analista financiero, quiero generar un reporte custom sobre un rango de fechas, revisar los accountGroups y descargar el resumen en .txt para enviarlo a legacy.

Simple (1-2 pts): búsqueda + selección
Medio (3-5 pts): nueva transacción + validación
Complejo (5-8 pts): reporte multi-fecha

⚡ Factores de aceleración

  • Hooks compartidos: todos usan useMutation y apiClient, por lo que extender la lógica (p.ej. nuevos reportes) solo implica nuevos DTOs y endpoints.
  • MSW: proporciona validaciones, IDs incrementales y errores (400/500) idénticos al backend, permitiendo probar confirmaciones Y/N y navegación sin backend real.
  • SystemHeader + KeyboardShortcuts: ya manejan la visualización del ID, nombre de programa y atajos (ENTER, F3, F5/F7/F8), manteniendo experiencia consistente.
  • Componentes reutilizados: LoadingSpinner, Alert y Chip ya están listos para indicar estados y mensajes provenientes de los endpoints.

🔧 Fundamento técnico

TransactionAddScreen.tsx

Formulario dividido en datos y confirmación, incluye ayuda visual, botones para copiar última transacción y atajos para limpiar o aceptar el registro.

Hook: useTransactionAdd

TransactionListScreen.tsx

Tabla enriquecida con columnas de fecha, descripción y monto; selecciona transacción y activa navegación con F7/F8.

Hook: useTransactionList (maneja first/lastTransactionId, hasNext/Previous y llamados a /list, /next-page, /previous-page).

TransactionViewScreen.tsx

Consulta por Transaction ID y limpia el estado con /api/transaction/clear, mostrando merchant, timestamps y encabezados de legacy.

Hook: useTransactionView

TransactionReportsScreen.tsx

Genera reportes mensual/anual/custom con confirmación Y/N y descarga simulada tipo PDF (.txt) vía TransactionReportTable.

Hook: useTransactionReports (maneja accountGroups, reportType, jobId y errores).

apiClient / useMutation

Centraliza headers JSON + Authorization, timeout 10 s y detección de respuestas MSW/real; `useMutation` transforma errores en Error para mostrar en UI.

SystemHeader

Muestra transactionId, programName y los atajos (F3/F5/F7/F8/ENTER) que todos los screens usan para regresar o confirmar.

🌐 Interfaces públicas

POST /api/transactions                   → Registra TransactionAddRequest (accountId/cardNumber, type/cat/source/description, amount, dates, merchant, confirmation Y/N) y responde con success/transactionId/message.
POST /api/transactions/list              → Lista inicial y búsquedas con transactionId/pageNumber, devuelve transactions[], currentPage, hasNextPage/hasPreviousPage, first/lastTransactionId.
POST /api/transactions/next-page         → Pide la página siguiente usando lastTransactionId.
POST /api/transactions/previous-page     → Retrocede usando firstTransactionId.
GET  /api/transaction-view/search        → Obtiene merchant, amount, timestamps y encabezados (currentDate, programName) por transactionId.
POST /api/transaction/clear              → Resetea la vista con template vacío e infoMessage.
POST /api/v1/reports/transactions/monthly→ Reporte mensual con reportData (accountGroups, totals) y grandTotal.
POST /api/v1/reports/transactions/yearly → Igual para anual.
POST /api/v1/reports/transactions/custom → Incluye startDate/endDate, confirmed: true y responde con accountGroups, grandTotal, accountCount, totalTransactionCount.

📊 Modelos y validaciones clave

Modelo Descripción Regla clave
TransactionAddRequestCampos de account/card, type/cat/source/description, amount, dates y merchant.confirmation debe ser 'Y' y amount está en ±99,999,999.99; las fechas no pueden estar vacías.
TransactionListResponseTransacciones con ID, fecha, descripción, monto, paginación y marcadores first/last.Transaction ID solo numérico y require S para navegar.
TransactionViewResponseDetalle de transacción con merchant, timestamps y encabezados de legacy.No se permite Transaction ID vacío y se valida con regex numérico.
TransactionReportResponsereportData con accountGroups, grandTotal, accountCount, totalTransactionCount.confirmed: true y custom requiere startDate ≤ endDate.

📋 Reglas de negocio confirmadas

RN-TX-01: Solo se puede seleccionar una transacción con la acción S; cualquier otra tecla devuelve «Invalid selection. Valid value is S». (useTransactionList.ts:188)
RN-TX-02: Cada registro o reporte requiere confirmación Y; un N cancela la operación con una alerta contextual.
RN-TX-03: transactionTypeCode y transactionCategoryCode deben ser numéricos (máx. 2 y 4 dígitos) y amount en ±99,999,999.99; los merchant datos no pueden estar incompletos.
RN-TX-04: Las búsquedas de Transaction ID validan ^\d+$ y no permiten cadenas vacías; mensajes reflejan el backend legacy.
RN-TX-05: Reportes custom no aceptan startDate > endDate; la validación ocurre antes de llamar al backend. (useTransactionReports.ts:106)

✅ Patrones de criterios de aceptación

Validación de monto

Dado un transactionAmount no numérico
Cuando intento avanzar
Entonces aparece «Amount should be in format -99999999.99» y se bloquea el envío.

Navegación paginada

Dado más páginas disponibles
Cuando presiono F8
Entonces el hook llama a /transactions/next-page con lastTransactionId.

Reporte custom

Dado el modo Custom y confirmación Y
Cuando presiono Generate
Entonces se llama a /v1/reports/transactions/custom con startDate/endDate y se muestra el resumen con accountGroups.

Consulta protegida

Dado un Token válido
Cuando accedo a TransactionViewScreen
Entonces SystemHeader muestra programName/transactionId y los datos del merchant.

⚡ Presupuestos de performance

  • Las pantallas usan LoadingSpinner y tractores de estado en trasfondo, esperando respuestas de MSW/back en < 1 s para mantener la fluidez del teclado heroico.
  • apiClient aplica timeout de 10 s y aborta peticiones anteriores cuando se navega rápidamente.
  • Los hooks abortan mutaciones previas para evitar resultados obsoletos (importante en búsquedas y paginación).

🚧 Dependencias & readiness

SystemHeader + Layout compartido

Todos los screens inyectan SystemHeader para mostrar encabezados legacy y habilitar rutas protegidas.

apiClient + useMutation

Encapsulan lógica de cabeceras, timeouts, detección de MSW vs backend y manejo de errores en UI.

Atajos de teclado

F3/Escape sale del módulo, F5 y ENTER confirman, F7/F8 controlan paginación de la lista, igual que los operadores heredados.

MSW

Permite validar la integridad de los flujos sin backend real, pero se recomienda revisar contratos reales al integrar con servicios productivos.
