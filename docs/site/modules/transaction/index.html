💸 TRANSACTION - Guía Operacional

Registro, búsqueda y reportes con las reglas de validación del backend heredado, atajos de teclado (F3/F5/F7/F8) y el componente SystemHeader como marco.

Documentación extendida: docs/modules/transaction/transaction-overview.md

📋 Visión general del módulo

TRANSACTION coordina cuatro pantallas protegidas (alta, listados paginados, vista detallada y reportes) que comparten hooks especializados y validaciones replicadas del flujo legacy. Los usuarios confían en una confirmación explícita Y/N antes de enviar transacciones o reportes y envían acciones con atajos heredados.

  • Alta dual-step con verificación de merchant, tipo/categoría y límite en ±99,999,999.99 previo a POST /api/transactions.
  • Lista paginada que mantiene firstTransactionId, lastTransactionId y permite selección con la tecla S antes de navegar al detalle.
  • Vista de transacción que limpia con POST /api/transaction/clear y muestra header + datos del merchant/monto/timestamps.
  • Reportes mensual/anual/custom con confirmación y descarga simulada (.txt) mostrando accountGroups.

🎯 Guía de desarrollo de Historias de Usuario

📝 Plantillas específicas

  • Operador Back-office: Como operador, quiero registrar manualmente una transacción identificando account/card, monto y merchant para corregir un ajuste legado antes de cerrar el ciclo.
  • Supervisor de auditoría: Como auditor, quiero buscar un Transaction ID numérico, navegar con F7/F8 y seleccionar la fila con S para revisar los metadatos antes de aprobar el cambio.
  • Analista financiero: Como analista, quiero generar un reporte custom (inicio/fin) y confirmar con Y para descargar el resumen y compartirlo con legacy.

⚡ Factores de aceleración

  • Hooks compartidos: useTransactionAdd, useTransactionList, useTransactionView y useTransactionReports extienden useMutation y apiClient, por lo que basta con duplicar estados o validaciones puntuales.
  • SystemHeader & atajos: la envoltura ya expone transactionId, programName y las teclas F3/F5/F7/F8/ENTER, lo cual mantiene consistencia sin reimplementar.
  • MSW handlers: `transaction*.ts` simulan errores y validaciones (montos, códigos, confirmación) para pruebas offline.

📊 Guía de complejidad

  • Simple (1-2 pts) Validar y enviar transacciones ya registradas con campos precompletados y confirmación Y.
  • Media (3-5 pts) Agregar nuevas validaciones de merchant/categoría y mostrar mensajes contextuales antes de la confirmación Y.
  • Compleja (5-8 pts) Extender los reportes custom con cálculos adicionales, nuevas columnas de accountGroups y la descarga tipo `.txt`.

🔧 Fundamento técnico

🏗️ Componentes clave

TransactionAddScreen

Formulario multi-step (datos → revisión) con validaciones de account/card, tipo/categoría, monto y merchant; solicita confirmación Y antes de llamar POST /api/transactions.

TransactionListScreen

Tabla paginada con columnas de fecha, descripción y monto; administra campos firstTransactionId y lastTransactionId y respeta atajos F7/F8/S.

TransactionViewScreen / Reports

Vista detallada y reportes que limpian el estado con /api/transaction/clear, muestran merchant + timestamps y exponen descarga de reportes (.txt) para legacy.

🔗 Dependencias

  • SystemHeader: muestra metadatos de transacción y expone atajos.
  • apiClient + useMutation: base para todos los hooks (app/services/api.ts y app/hooks/useApi.ts).
  • MSW handlers: transactionAddHandlers, transactionListHandlers, transactionViewHandlers, transactionReportsHandlers replican validaciones.
  • LoadingSpinner / Alert: muestran estados de carga y errores en cada pantalla.

🌐 Interfaces públicas

POST /api/transactions            - Alta con account/card, tipo/categoría, monto, merchant y confirmación Y
POST /api/transactions/list       - Lista inicial y búsquedas (usa transactionId, pageNumber)
POST /api/transactions/next-page  - Avance (F8) con lastTransactionId
POST /api/transactions/previous-page - Retroceso (F7) con firstTransactionId
GET  /api/transaction-view/search - Detalle por Transaction ID (merchant, timestamps, errores)
POST /api/transaction/clear       - Limpia la vista antes de nueva búsqueda
POST /api/v1/reports/transactions/monthly - Reporte mensual con confirmación
POST /api/v1/reports/transactions/yearly  - Reporte anual
POST /api/v1/reports/transactions/custom  - Reporte custom (startDate, endDate, confirmed)

🧾 Criterios de aceptación

  • Validación: Si `transactionAmount` tiene texto, el formulario muestra `Amount should be in format -99999999.99` y no llama al backend.
  • Interacción: Presionar F8 en la lista invoca `/transactions/next-page` con el último ID antes de actualizar la tabla.
  • Reporte: Confirmar `Custom + Y` dispara `/v1/reports/transactions/custom` y presenta `accountGroups` más totales.
